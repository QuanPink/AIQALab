Find, fix, verify, and guard against regression: $ARGUMENTS

> This command follows rules defined in /shared-qa-rules. All rules in that file are mandatory.

---

## Purpose

This command runs a complete Find-Fix-Verify cycle in one pass. It combines failure analysis, code fix, re-run verification, and regression test generation into a single workflow. Use when you want the fastest path from "test failed" to "test fixed + guarded."

---

## Phase 0 — Input

| Input | Action |
|-------|--------|
| `<TestClass>#<method>` | Fix specific test failure |
| `<TestClass>` | Fix all failures in class |
| `all` | Fix all recent failures |
| Blank | Read latest failure from `.claude/artifacts/failure-analysis/latest.md` or from surefire reports |

---

## Phase 1 — Find (Analyze)

Execute the same logic as `/qa-analyze-failure`:

1. Read `target/surefire-reports/*.xml` — find failed tests
2. Read test source code — identify exact failing line
3. Classify root cause (Locator, Assertion, Timing, Auth, Data, Service Layer, Null-by-design, Public Endpoint)

Output per failure:
```
FOUND: <TestClass>#<method>
  Category: <root cause>
  Line: <file>:<line>
  Cause: <brief explanation>
```

---

## Phase 2 — Fix (Apply)

Execute the same logic as `/qa-fix-test`:

1. Apply fix based on category (Locator → update By selector, Assertion → update expected value, Service → fix service class, etc.)
2. Edit actual source files using Edit tool

Output per fix:
```
FIXED: <TestClass>#<method>
  File: <path>
  Change: <what changed>
  Confidence: High / Medium / Low
```

---

## Phase 3 — Verify (Re-run)

Immediately run the fixed test to confirm:
```bash
mvn test -Denv=dev -Dtest=<TestClass>#<method> -Dmaven.test.failure.ignore=false
```

| Result | Action |
|--------|--------|
| PASS | Proceed to Phase 4 |
| FAIL (same error) | Fix did not work — report and stop. Suggest manual investigation. |
| FAIL (different error) | Fix introduced new issue — revert and report. |

Output:
```
VERIFIED: <TestClass>#<method>
  Result: PASS ✅
  Duration: <X>s
```

If confidence was Medium/Low, run 3 times to check flakiness:
```bash
mvn test -Denv=dev -Dtest=<TestClass>#<method> -Dsurefire.rerunFailingTestsCount=3
```

---

## Phase 4 — Guard (Generate Regression Test)

After verification passes, auto-generate a regression test:
```java
@Test(groups = {"regression"}, description = "Regression — <brief failure description>")
@Story("<Feature> — Regression Guard")
@Severity(SeverityLevel.CRITICAL)
public void shouldNotRegress_<originalMethodName>() {
    // Regression guard for: <TestClass>#<method> — <date>
    // Original failure: <root cause category> — <brief description>
    
    // Reproduce exact conditions
    // Assert the fix holds
}
```

Add the regression test to the same test class, at the bottom, in a clearly marked section:
```java
// ═══════════════════════════════════════════
// REGRESSION GUARDS — auto-generated by /qa-fix-and-verify
// ═══════════════════════════════════════════
```

## Rules

1. MUST run verification (Phase 3) — never skip
2. MUST generate regression test (Phase 4) for every successful fix
3. If fix confidence is Low, STOP after Phase 2 and ask user before running verify
4. If verify fails, REVERT the fix and report — do not attempt a second fix automatically
5. Regression test goes in the SAME test class, not a separate file
6. This command modifies code — use with caution. Review changes with `/qa-review-test` after completion

---

## Anti-patterns (STRICTLY FORBIDDEN)

| ❌ Forbidden | ✔ Required |
|-------------|-----------|
| Fix without verify | Always run the test after fixing |
| Verify without regression guard | Always generate regression test after fix passes |
| Multiple fix attempts in one cycle | One fix attempt → verify → if fail, stop and report |
| Regression test in separate file | Same test class, clearly marked section |
| Skipping low-confidence fixes silently | Ask user before proceeding with low-confidence fix |
