Review and audit the following test file(s) for quality compliance: $ARGUMENTS

> This command follows rules defined in /shared-qa-rules. All rules in that file are mandatory.

---

## Purpose

This command acts as a **quality gate** for test code — whether generated by other QA commands or written manually. It audits test files against the established standards and produces an actionable compliance report.

---

## Phase 0 — Input Resolution

1. If `$ARGUMENTS` is a file path → read and audit that file
2. If `$ARGUMENTS` is a directory path → scan all `*Test.java` files recursively
3. If `$ARGUMENTS` is a class name → locate it under `src/test/java/` and audit
4. If `$ARGUMENTS` is blank → prompt user for target

For each file found, run Phase 1–5 below independently.

---

## Phase 1 — Architecture Compliance

Check the layered architecture rule: `Test → Service → ApiClient → Params`

### 1.1 Service Layer Usage

| Check | Pass | Fail |
|-------|------|------|
| Test calls Service methods (not ApiClient directly) | `service.searchByKeyword(...)` | `apiClient.call(...)` or `apiClient.get(...)` |
| Service class exists for the business flow | File found in `src/main/java/com/framework/api/service/` | No service class — test builds request inline |
| Service contains NO assertions | No `assert*`, `assertThat`, `assertEquals` in service | Assertions found in service class |
| Service contains NO test logic | No `@Test`, `@DataProvider`, conditional test branching | Test annotations or branching found in service |

### 1.2 ApiClient Usage

| Check | Pass | Fail |
|-------|------|------|
| ApiClient class exists | File found in `src/main/java/com/framework/api/client/` | Missing — requests built ad-hoc in test |
| ApiClient method naming follows business intent | `search(params)`, `findById(id)`, `create(request)` | `callEndpoint()`, `postV2()`, `filterV1()` |

### 1.3 File Location

| Artifact | Correct Path | Common Mistake |
|----------|-------------|----------------|
| Test class | `src/test/java/com/framework/...` | — |
| Service class | `src/main/java/com/framework/api/service/` | Placed in `src/test/java/` |
| ApiClient | `src/main/java/com/framework/api/client/` | Placed in `src/test/java/` |
| Page Objects | `src/main/java/com/framework/ui/pages/` | Placed in `src/test/java/` |

## Phase 2 — Naming Convention Audit

### 2.1 Class Naming

Expected format: `<Verb><BusinessEntity>[Qualifier]ApiTest` or `<BusinessFlow>Test`

| Check | Pass | Fail |
|-------|------|------|
| Class name starts with business verb | `SearchUserApiTest`, `CreateOrderApiTest` | `PositionStatisticSearchApiTest` |
| No endpoint path in name | `GetTraderListApiTest` | `ApiV2PositionFilterTest` |
| No technical field names | `SearchOrderApiTest` | `TxHashLookupTest` |

### 2.2 Method Naming

Expected pattern: `should<Behavior>[When<Condition>]()`

| Check | Pass | Fail |
|-------|------|------|
| Starts with `should` | `shouldReturnResultsForValidInput()` | `testSearch()`, `test001()` |
| Describes behavior, not implementation | `shouldReturnEmptyWhenNoDataFound()` | `testGetEndpoint()` |
| No sequential IDs | `shouldFilterByDateRange()` | `TC001_search()`, `test_01()` |

### 2.3 DataProvider Naming

| Check | Pass | Fail |
|-------|------|------|
| Named by behavior | `validFilterCombinations`, `knownIdentifiers` | `statusProvider`, `dataList`, `ids` |

### 2.4 Service Method Naming

| Check | Pass | Fail |
|-------|------|------|
| Follows business intent | `searchByKeyword()`, `findById()`, `createOrder()` | `filterV2()`, `callEndpointX()`, `executeRequest()` |

## Phase 3 — Test Quality Audit

### 3.1 Test Grouping

Check that tests are organized into the 5 standard groups:

| Group | Present | Missing Impact |
|-------|---------|----------------|
| Positive | Required | Core happy path untested |
| Negative | Required | Error handling untested |
| Edge Cases | Required | Boundary conditions untested |
| Cross-cutting | Required | Auth, pagination, schema untested |
| Security | Conditional | Only if auth is applicable |

### 3.2 Assertion Quality

| Check | Pass | Fail |
|-------|------|------|
| Assertions only in test class | All `assert*` in `*Test.java` | Assertions in Service or ApiClient |
| Specific assertions used | `assertEquals(expected, actual)`, `assertThat(...).isEqualTo(...)` | Only `assertNotNull(response)` |
| Response schema validated | Status code + body structure checked | Only status code checked |
| Business logic validated | Result values match input/expected | Only checks "not empty" |
| No scattered assertions | Related checks grouped in one test method | Same field asserted in multiple methods without purpose |

### 3.3 Test Data

| Check | Pass | Fail |
|-------|------|------|
| Constants use generic naming | `VALID_INPUT`, `KNOWN_ID`, `SAMPLE_VALUE` | `COPIN_WALLET`, `BTC_ADDRESS` |
| Fallback pattern used for lookup | `System.getProperty("test.entity.id", "fallback")` | Hardcoded ID with no fallback |
| No domain-specific format assumptions | Treats identifiers as opaque strings | Validates format like `0x...` or specific length |
| Fallback triggers "not found" test | When default fallback is used, test validates not-found behavior | Fallback value used but no not-found assertion |

### 3.4 Allure Annotations

| Check | Pass | Fail |
|-------|------|------|
| `@Epic` at class level | Present | Missing |
| `@Feature` at class level | Present | Missing |
| `@Story` at method level | Present on each test method | Missing or inconsistent |
| `@Severity` at method level | Present on each test method | Missing |

## Phase 4 — Cross-API Isolation

| Check | Pass | Fail |
|-------|------|------|
| One API per test class | Class tests single endpoint/business action | Multiple endpoints tested in one class |
| No shared logic across APIs | Each test class is self-contained | Shared helper calls different APIs |
| Service class scoped to one flow | `SearchUserService` handles search only | `UserService` handles search + create + delete |

## Phase 5 — Common Anti-patterns Detection

Scan for these specific violations:

| Anti-pattern | How to Detect | Severity |
|-------------|---------------|----------|
| Direct ApiClient call in test | `apiClient.` appears in test method body | 🔴 Critical |
| Endpoint path in naming | Class/method contains `/api/`, `/v1/`, `/v2/` | 🔴 Critical |
| Domain hardcoding | `copin`, `binance`, `dex`, protocol-specific terms in test code | 🔴 Critical |
| Sequential test IDs | Method name contains `TC001`, `test_01`, `S001` | 🟡 High |
| Missing service layer | Test constructs request params inline | 🟡 High |
| Thread.sleep in test | `Thread.sleep(` found | 🟡 High |
| Assertions in service | `assert` keyword in service class | 🟡 High |
| Production code in test dir | Service/ApiClient/PageObject under `src/test/java/` | 🟡 High |
| Only status code assertion | `assertEquals(200, ...)` with no body validation | 🟢 Medium |
| Missing Allure annotations | No `@Epic`, `@Feature`, `@Story`, `@Severity` | 🟢 Medium |
| Overly broad DataProvider name | `data`, `provider`, `list` without behavior context | 🟢 Medium |

### 5.1 Full Surface Scan Rule (MANDATORY)

When a domain-specific term OR technical field name is detected in ANY location within a file, the review MUST scan ALL 9 surface areas below in that same file and report ALL occurrences in a single pass. DO NOT report only the most visible occurrences — report every single one.

| # | Surface Area | Example location |
|---|-------------|-----------------|
| 1 | Constant names | `private static final String SOLANA_PREFIX = ...` |
| 2 | Constant values | `= List.of("DYDX", "JUPITER")` |
| 3 | Method names | `shouldFindTradersWithEvmAddressPrefix()` |
| 4 | @Story annotations | `@Story("Search Trader — EVM Address")` |
| 5 | @Test(description) | `@Test(description = "EVM address prefix — returns...")` |
| 6 | @Epic / @Feature | `@Epic("copin.io API")` |
| 7 | DataProvider descriptions | `{"0xa", "EVM lowercase"}` |
| 8 | Assertion messages (.as) | `.as("Solana accounts must start with...")` |
| 9 | Javadoc / inline comments | `/** Searches for dYdX traders */` |

### 5.2 Domain Term vs Format Term Distinction (MANDATORY)

When scanning for domain-specific terms, the review MUST distinguish between:

**Domain terms (MUST flag as violation):**
Terms tied to a specific business, protocol, or ecosystem. These bind code to a particular domain.

| Term type | Examples | Why flag |
|-----------|----------|----------|
| Protocol names | EVM, Solana, dYdX, JUPITER, GMX, KWENTA | Ties code to specific blockchain protocol |
| Platform names | copin, binance, dex, uniswap | Ties code to specific product |
| Network names | Ethereum, Arbitrum, Polygon | Ties code to specific network |
| API field names as identifiers | TxHash (as method name), txHash (as constant name) | Ties naming to backend implementation |

**Format terms (DO NOT flag — these are acceptable):**
Standard technical encoding or format names. These describe data format, not business domain.

| Term type | Examples | Why acceptable |
|-----------|----------|---------------|
| Encoding formats | Hex, Base58, Base64, Bech32, UTF-8 | Industry-standard format names |
| Data formats | JSON, XML, CSV, GraphQL | Standard data interchange formats |
| Hash formats | SHA256, MD5, Keccak | Standard cryptographic names |
| Number formats | BigDecimal, Long, Double | Standard type names |

**Decision rule:**
- "Solana address" → 🔴 domain term (flag it)
- "Base58 address" → ✅ format term (same encoding, domain-agnostic name)
- "dYdX prefix" → 🔴 domain term (flag it)
- "Bech32 prefix" → ✅ format term (standard encoding name)
- "EVM lowercase" → 🔴 domain term (flag it)
- "hex lowercase" → ✅ format term (standard encoding name)

**Constant naming examples:**

| ❌ Domain (flag) | ✅ Format (acceptable) |
|-----------------|----------------------|
| `SOLANA_PREFIX` | `BASE58_ACCOUNT_PREFIX` |
| `EVM_PREFIX` | `HEX_ACCOUNT_PREFIX` |
| `DYDX_PREFIX` | `BECH32_ACCOUNT_PREFIX` |
| `SOLANA_PROTOCOL_FILTER` | `BASE58_PROTOCOL_FILTER` |

DO NOT rename format-descriptive constants to generic labels like `KEYWORD_PREFIX_A` — this destroys code readability without improving domain-agnosticism.

**Scan procedure:**
1. Collect all domain-specific and technical field terms found in the file (e.g., "EVM", "Solana", "dYdX", "DYDX", "JUPITER", "TxHash", "txHash")
2. For EACH term, scan ALL 9 surface areas
3. Report every occurrence grouped by term, not by surface area
4. The fix recommendation MUST list ALL locations for each term so they can be fixed in one pass

**Example output format:**

```
[🔴 Critical] Domain term "EVM" found in 4 surface areas:
  → Method name: shouldFindTradersWithEvmAddressPrefix() (line 66)
  → @Story: "Search Trader — EVM Address" (line 65)
  → @Test(description): "EVM address prefix — returns..." (line 66)
  → DataProvider: {"0xa", "EVM lowercase"} (line 152)
  → Fix ALL: Replace "EVM" with "Hex" in all 4 locations
```

This prevents iterative review-fix-review loops where each round discovers new occurrences of the same term in different surface areas.

## Phase 6 — Output

Generate a compliance report with the following structure:

### 6.1 Summary Table (per file)

```
| File | Architecture | Naming | Test Quality | Isolation | Anti-patterns | Overall |
|------|-------------|--------|--------------|-----------|---------------|---------|
| SearchUserApiTest.java | ✅ | ✅ | ⚠️ | ✅ | ✅ | PASS |
| CreateOrderApiTest.java | ❌ | ⚠️ | ❌ | ✅ | ❌ | FAIL |
```

Scoring:
- ✅ = All checks pass
- ⚠️ = Minor issues (🟢 severity only)
- ❌ = Critical or high severity issues found
- Overall: PASS (no ❌), WARN (no 🔴 but has 🟡), FAIL (has 🔴)

### 6.2 Issue List (per file)

For each issue found:

```
[🔴 Critical] SearchOrderApiTest.java
  → Phase 1.1: Test calls apiClient.search() directly at line 45
  → Fix: Create SearchOrderService with method searchByKeyword(), move request construction there
```

### 6.3 Recommended Actions

Group by priority:

1. **Fix immediately (🔴):** List all critical violations with specific fix instructions
2. **Fix soon (🟡):** List all high severity issues
3. **Improve (🟢):** List all medium severity suggestions

### 6.4 Refactor Suggestions

If the file needs significant restructuring, provide:
- Current structure (as-is)
- Target structure (to-be)
- Step-by-step refactor plan
- Reference to relevant command: `/qa-generate-api-test` or `/qa-generate-ui-test`

## Anti-patterns (STRICTLY FORBIDDEN)

| ❌ Forbidden | ✔ Required |
|-------------|-----------|
| Auto-fixing code during review | Report and recommend only — never auto-fix |
| Judging business logic correctness | Structural compliance only |
| Reviewing based on assumptions | Always reference current shared-qa-rules |
| Reporting issues without fix suggestions | Every issue must include a concrete fix |
| Single-pass domain scan (only most visible terms) | Full 9-surface-area scan for every detected domain term |
| Skipping architecture check if naming looks clean | Always verify Service Layer exists between test and ApiClient |
